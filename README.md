Опис проекту

У цьому проекті я працював з набором даних MovieLens та графовою базою даних Neo4j. Основною метою роботи було побудувати граф користувачів, фільмів та жанрів, завантажити дані в Neo4j, виконати запити Cypher різної складності, знайти супервузли та застосувати алгоритми Graph Data Science.

Для роботи використовувалася база даних Neo4j AuraDB. Спочатку база даних створювалася на плані AuraDB Free, але під час виконання частини GDS виникло обмеження: безкоштовний план не підтримував необхідні проєкції GDS. Тому для запуску PageRank, Louvain та найкоротшого шляху була створена нова база даних у пробній версії AuraDB Professional з увімкненим плагіном GDS.

Графова модель даних

У графі використовуються такі типи вузлів:

User
Movie
Genre

Вузол `User` зберігає інформацію про користувача:

userId
gender
age
occupation

Вузол `Movie` зберігає інформацію про фільм:

movieId
title
year

Вузол `Genre` зберігає назву жанру:

name

Між вузлами створюються такі зв’язки:

(User)-[:RATED]->(Movie)
(Movie)-[:HAS_GENRE]->(Genre)

Зв’язок `RATED` має властивості:

rating
timestamp

Така модель добре підходить для MovieLens, тому що користувачі, фільми та жанри природно утворюють граф. Завдяки цьому можна зручно шукати спільні фільми, схожих користувачів, рекомендації, популярні жанри та шляхи між користувачами.

Завантаження даних

Для завантаження даних використовується файл:

queries/part2_load.cypher

Перед завантаженням були створені constraints для унікальності користувачів, фільмів і жанрів:

CREATE CONSTRAINT user_id_unique IF NOT EXISTS
FOR (u:User)
REQUIRE u.userId IS UNIQUE;

CREATE CONSTRAINT movie_id_unique IF NOT EXISTS
FOR (m:Movie)
REQUIRE m.movieId IS UNIQUE;

CREATE CONSTRAINT genre_name_unique IF NOT EXISTS
FOR (g:Genre)
REQUIRE g.name IS UNIQUE;

Дані завантажувалися через `LOAD CSV WITH HEADERS` із GitHub raw links.

Для зв’язків `RATED` використовувався `apoc.periodic.iterate`, тому що ratings-файл містить велику кількість оцінок. Через обмеження безкоштовного AuraDB для зв’язків `RATED` було використано підмножину з перших `200000` оцінок.

Після завантаження база містила:

users: 6040
movies: 3883
genres: 18
genreLinks: 6408
ratings: 200000

Вузли `User`, `Movie`, `Genre` і зв’язки `HAS_GENRE` були завантажені повністю. Зв’язки `RATED` були завантажені як підмножина, щоб уникнути перевищення лімітів AuraDB.

Частина 3. Cypher-запити

Файл із запитами:

queries/part3.cypher

У цій частині були виконані 6 Cypher-запитів різної складності.

Запит 1. Фільми жанру Thriller із середнім рейтингом вище 4.0

Запит знаходить фільми жанру `Thriller`, рахує середній рейтинг і залишає тільки ті фільми, у яких середня оцінка вище 4.0. Також використовується фільтр `ratingCount >= 20`, щоб у результат не потрапляли фільми з дуже малою кількістю оцінок.

Цей запит показує, які Thriller-фільми стабільно отримували високі оцінки.

Запит 2. Користувачі, які поставили оцінку 5 більш ніж 50 фільмам

Цей запит шукає користувачів, які часто ставили максимальну оцінку. Він дозволяє знайти найбільш “щедрих” або дуже активних користувачів.

Запит 3. Фільми, які два користувачі оцінили високо

Запит шукає фільми, які два конкретні користувачі оцінили на 4 або 5. Це корисно для порівняння смаків двох користувачів.

Запит 4. Жанри з високим середнім рейтингом

Запит групує фільми за жанрами, рахує середній рейтинг і кількість оцінок. Фільтр `ratingCount >= 1000` допомагає залишити тільки жанри з достатньою кількістю оцінок.

Запит 5. Рекомендації через схожих користувачів

Цей запит реалізує просту рекомендаційну логіку. Спочатку знаходяться користувачі, які мають спільні високо оцінені фільми з цільовим користувачем. Потім система рекомендує фільми, які сподобались схожим користувачам, але які цільовий користувач ще не оцінював.

Це один із прикладів, де графова модель виглядає дуже природно, тому що логіка рекомендації описується через шлях у графі.

Запит 6. Найкоротший шлях між двома користувачами

Запит шукає шлях між двома користувачами через спільні фільми та оцінки. Такий тип запиту було б складніше реалізувати в реляційній базі, тому що довелося б вручну будувати багато JOIN-зв’язків.

У Neo4j така задача описується природніше, тому що користувачі й фільми вже з’єднані ребрами.

Частина 4. Супервузли

Файл із запитами:

queries/part4_supernodes.cypher

Супервузол у графі це вузол, який має дуже багато зв’язків. У цьому датасеті потенційними Супервузлами можуть бути популярні фільми, великі жанри та дуже активні користувачі.

Найпопулярніші фільми за кількістю оцінок

Серед фільмів найбільше зв’язків `RATED` отримали:

American Beauty (1999): 731 оцінка
Star Wars: Episode V - The Empire Strikes Back (1980): 605 оцінок
Jurassic Park (1993): 604 оцінки
Star Wars: Episode IV - A New Hope (1977): 597 оцінок
Star Wars: Episode VI - Return of the Jedi (1983): 594 оцінки

Ці фільми можна вважати Супервузлами серед `Movie`, тому що до них веде багато зв’язків від користувачів.

Найбільші жанри

Серед жанрів найбільшими Супервузлами стали:

Drama: 1603 зв’язки
Comedy: 1200 зв’язків

Жанри часто стають Супервузлами, тому що один жанр може бути пов’язаний із великою кількістю фільмів.

Найактивніші користувачі

Серед користувачів були знайдені дуже активні вузли, наприклад:

User 1181: 1521 зв’язок
User 889: 1518 зв’язків
User 1150: 1302 зв’язки

Такі користувачі поставили багато оцінок, тому їхні вузли мають велику кількість вихідних ребер `RATED`.

Чому Супервузли можуть бути проблемою

Запити, які проходять через Супервузли, можуть працювати повільніше. Причина в тому, що Neo4j потрібно обійти велику кількість зв’язків. Наприклад, якщо запит проходить через жанр `Drama`, він може зачепити понад 1600 фільмів.

Для зменшення впливу Супервузлів можна:
додавати фільтри за rating, genre або ratingCount
обмежувати глибину обходу графа
використовувати LIMIT
створювати індекси для стартових вузлів
будувати окремі GDS-проекції для важких аналітичних задач

5.1 PageRank

Для PageRank був створений граф фільмів. Два фільми з’єднувалися ребром `CO_RATED`, якщо багато користувачів високо оцінили обидва фільми.

Було створено:

CO_RATED relationships: 50000

Після створення GDS-проекції `movieGraph` було отримано:

nodeCount: 3883
relationshipCount: 100000

Результати PageRank:

1. American Beauty (1999): 9.3539
2. Star Wars: Episode V - The Empire Strikes Back (1980): 8.7791
3. Star Wars: Episode IV - A New Hope (1977): 8.4681
4. Raiders of the Lost Ark (1981): 7.5403
5. Matrix, The (1999): 7.1179
6. Fargo (1996): 6.7849
7. Saving Private Ryan (1998): 6.5475
8. Silence of the Lambs, The (1991): 6.4284

Високий PageRank у цьому графі означає, що фільм займає центральне місце в мережі схожих фільмів. Це не просто фільм із великою кількістю оцінок. Важливо також те, з якими іншими фільмами він пов’язаний і наскільки сильні ці зв’язки.

5.2 Louvain

Для Louvain був створений граф схожості користувачів. Два користувачі з’єднувалися ребром `SIMILAR`, якщо вони високо оцінили однакові фільми. Вага ребра показує кількість спільних фільмів, які обом користувачам сподобались.

Алгоритм Louvain знайшов одну велику основну спільноту:

communityId 1126: 724 users

Інші спільноти були значно меншими, часто по одному користувачу. Це можна пояснити тим, що для роботи використовувалося підмноження з 200000 оцінок, тому не всі користувачі мали достатньо спільних високо оцінених фільмів для формування великих кластерів.

Для перевірки змісту спільнот були проаналізовані найпопулярніші жанри всередині кластерів. Серед топ жанрів зустрічалися:

Adventure
Comedy
Action
Sci-Fi
Drama

Це показує, що Louvain справді групує користувачів за схожістю їхніх оцінок і жанрових вподобань.

5.3 Shortest Path

Для пошуку найкоротшого шляху між користувачами використовувався алгоритм Dijkstra на графі `SIMILAR`.

Результат:

totalCost: 65.0
userPath: [424, 845, 1015]
pathLength: 2

Це означає, що між користувачами `424` і `1015` був знайдений шлях через проміжного користувача `845`.

У цьому графі `weight` показує силу схожості між користувачами. Для Dijkstra це не ідеальна вага, тому що алгоритм шукає мінімальну вартість шляху. У більш коректній моделі можна було б створити окрему властивість:

cost = 1 / weight

і запускати shortest path саме по `cost`. У цій роботі алгоритм використано для демонстрації пошуку шляху між користувачами у графі схожості.

Частина 6. Аналіз і висновки

Файл з аналізом:

queries/part6_analysis.md

1. Граф vs SQL

У цьому завданні графова модель була зручнішою для запитів, де потрібно рухатися через зв’язки між користувачами, фільмами та жанрами. Наприклад, запит із рекомендаціями для користувача було б складніше писати в SQL, тому що там потрібно було б робити кілька JOIN між таблицями `ratings`, `users` і `movies`.

У Cypher це виглядає природніше, бо можна описати шлях у графі: користувач оцінив фільм, інший користувач теж оцінив цей фільм, а потім цей інший користувач оцінив ще інші фільми.

Також графова модель добре підходить для пошуку зв’язку між двома користувачами. У SQL такий запит був би складним, особливо якщо потрібно шукати не тільки прямий збіг, а шлях через кілька проміжних користувачів або фільмів.

2. Де графова модель програє

Реляційна модель була б зручнішою для простих агрегованих звітів. Наприклад, якщо потрібно просто порахувати кількість користувачів, середній рейтинг по всіх фільмах, кількість оцінок за роками або зробити табличний експорт, SQL виглядає простіше і звичніше.

Такі задачі не потребують складного обходу зв’язків, тому графова модель тут не дає великої переваги.

3. Покращення схеми

Для покращення схеми можна було б додати більше властивостей до вузлів і ребер. Наприклад, для ребра `RATED` можна зберігати не тільки `rating` і `timestamp`, а й окремо рік або місяць оцінки. Це дозволило б швидше робити запити за періодами.

Також можна було б окремо виділити вузли `Occupation` для професій користувачів, а не зберігати `occupation` як просту властивість у `User`. Це було б корисно, якщо потрібно часто аналізувати зв’язки між професіями, жанрами і фільмами.

Ще одне можливе покращення, створити окремий тип зв’язку `LIKED` для оцінок 4 або 5. Це спростило б запити для рекомендацій і графових алгоритмів, тому що не потрібно було б кожного разу фільтрувати `RATED` за `rating >= 4`.

У частині з GDS також було видно, що для shortest path краще мати окрему властивість `cost`. У нашій схемі `weight` показує силу схожості між користувачами, але Dijkstra сприймає вагу як вартість шляху. Тому більш коректно було б додати `cost = 1 / weight` і запускати алгоритм найкоротшого шляху саме по `cost`.

---

Висновок

У цьому проєкті було побудовано графову модель MovieLens у Neo4j. Дані були перетворені з `.dat` у `.csv`, завантажені в Neo4j AuraDB, а потім використані для Cypher-запитів і графової аналітики.

Було створено граф із користувачів, фільмів і жанрів. Після завантаження база містила 6040 користувачів, 3883 фільми, 18 жанрів, 6408 зв’язків між фільмами та жанрами і 200000 зв’язків оцінок.

У роботі були виконані звичайні Cypher-запити, рекомендаційний запит, пошук Супервузлів і GDS-алгоритми PageRank, Louvain та Dijkstra shortest path.

Головний висновок: графова модель добре підходить для задач, де важливі зв’язки між об’єктами. Для MovieLens це особливо корисно, тому що користувачі, фільми, жанри й оцінки утворюють природну мережу. Завдяки цьому можна зручно аналізувати схожість користувачів, центральність фільмів, популярні жанри та рекомендації.
