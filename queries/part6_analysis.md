1. Граф vs SQL

У цьому завданні графова модель була зручнішою для запитів, де потрібно рухатися через зв’язки між користувачами, фільмами та жанрами. Наприклад, запит із рекомендаціями для користувача було б складніше писати в SQL, тому що там потрібно було б робити кілька JOIN між таблицями ratings, users і movies, окремо знаходити схожих користувачів, потім виключати вже переглянуті фільми і рахувати рекомендаційний score. У Cypher це виглядає природніше, бо ми просто описуємо шлях у графі: користувач оцінив фільм, інший користувач теж оцінив цей фільм, а потім цей інший користувач оцінив ще інші фільми.

Також графова модель добре підходить для пошуку зв’язку між двома користувачами. У SQL такий запит був би складним, особливо якщо потрібно шукати не тільки прямий збіг, а шлях через кілька проміжних користувачів або фільмів. У Neo4j це можна описати через shortest path або GDS-алгоритми.

2. Де графова модель програє

Реляційна модель була б зручнішою для простих агрегованих звітів. Наприклад, якщо потрібно просто порахувати кількість користувачів, середній рейтинг по всіх фільмах, кількість оцінок за роками або зробити табличний експорт, SQL виглядає простіше і звичніше. Такі задачі не потребують складного обходу зв’язків, тому графова модель тут не дає великої переваги.

Також SQL може бути зручнішим для регулярної звітності, де дані мають чітку табличну структуру і потрібно часто будувати однакові аналітичні таблиці. Наприклад, звіт “топ-10 жанрів за середнім рейтингом” можна легко зробити і в SQL через JOIN, GROUP BY і ORDER BY.

3. Покращення схеми

Для покращення схеми можна було б додати більше властивостей до вузлів і ребер. Наприклад, для ребра RATED можна зберігати не тільки rating і timestamp, а й додатково рік або місяць оцінки. Це дозволило б швидше робити запити за періодами без додаткового перетворення timestamp.

Також можна було б окремо виділити вузли Occupation для професій користувачів, а не зберігати occupation як просту властивість у User. Це було б корисно, якщо потрібно часто аналізувати зв’язки між професіями, жанрами і фільмами. Але для цього завдання occupation як властивість користувача є достатнім рішенням.

Ще одне можливе покращення, створити окремі ребра або властивості для сильних оцінок. Наприклад, якщо користувач поставив фільму 4 або 5, можна було б створювати окремий тип зв’язку LIKED. Це спростило б запити для рекомендацій і графових алгоритмів, тому що не потрібно було б кожного разу фільтрувати RATED за rating >= 4.

У частині з GDS також було видно, що для shortest path краще мати окрему властивість cost. У нашій схемі weight показує силу схожості між користувачами, але Dijkstra сприймає вагу як вартість шляху. Тому більш коректно було б додати cost = 1 / weight і запускати алгоритм найкоротшого шляху саме по cost.
